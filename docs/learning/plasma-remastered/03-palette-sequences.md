# Layer 3 — Palette Sequences

**Source:** `src/effects/plasma/effect.remastered.js`, lines 102–163 (GLSL palette functions), lines 19–46 (colour themes)
**Concepts:** procedural palettes, cosine curves, colour remapping matrix, drop transitions, hue rotation

## What This Layer Covers

The sine harmonics from Layer 1 produce a colour index (0–255) at each pixel.
This layer explains how that number becomes a visible colour:

- The `ptau` cosine curve — the building block for all palettes
- Three procedural palette functions (red, rainbow, grey)
- Drop transitions that switch between palette sequences
- The 3×3 colour remapping matrix that creates 21 themes from one palette
- Hue rotation and saturation boost for fine colour control

---

## From Index to Colour

The plasma formula outputs a number from 0 to 255 — a **colour index**. By
itself this is just a number. The palette is the function that maps each
number to an RGB colour. Different palettes make the same plasma pattern look
completely different: fiery red, neon rainbow, or subtle grey.

```
Index:  0  ──────────── 64 ──────────── 128 ──────────── 192 ──────────── 255
        │               │                │                │                │
        ▼               ▼                ▼                ▼                ▼
Red:    black           bright red       black            blue→magenta     black
Rainbow: black          red              purple           green→yellow     green+white
Grey:   dark            dark             grey             dark             dark
```

## The ptau Building Block

Every palette is built from a single curve called `ptau`:

```glsl
float ptau(float k) {
  return cos(clamp(k, 0.0, 128.0) * TAU / 128.0 + PI) * 31.0 + 32.0;
}
```

This is a **raised cosine** over the range 0–128:

```
ptau output:
63 ┤          ╭────╮
   │         ╱      ╲
32 ┤────────╱        ╲────────
   │       ╱          ╲
 1 ┤──────╱            ╲──────
   └──────────────────────────
   0     32    64    96   128
```

- At k = 0: ptau = 1 (near-black)
- At k = 64: ptau = 63 (maximum brightness)
- At k = 128: ptau = 1 (near-black again)

The `+ PI` offset flips the cosine so it starts low, peaks in the middle, and
returns low. Every colour channel in every palette is generated by calling
`ptau()` with a different input, producing smooth bump-shaped intensity curves.

## Palette 0: Red

The 256-entry colour space is divided into 4 segments of 64 indices each:

```glsl
vec3 palette0(float ci) {
  float t = ci / 256.0 * 4.0;
  float seg = floor(t);
  float f = fract(t);
  if (seg < 1.0) return vec3(ptau(f * 63.0), ptau(0.0), ptau(0.0));
  if (seg < 2.0) return vec3(ptau((1.0 - f) * 63.0), ptau(0.0), ptau(0.0));
  if (seg < 3.0) return vec3(ptau(0.0), ptau(0.0), ptau(f * 63.0));
  return vec3(ptau(f * 63.0), ptau(0.0), ptau((1.0 - f) * 63.0));
}
```

| Segment | Indices | Red | Green | Blue | Visual |
|---------|---------|-----|-------|------|--------|
| 0 | 0–63 | ramp up | dark | dark | Black → bright red |
| 1 | 64–127 | ramp down | dark | dark | Bright red → black |
| 2 | 128–191 | dark | dark | ramp up | Black → bright blue |
| 3 | 192–255 | ramp up | dark | ramp down | Magenta → red |

The `fract(t)` gives a 0–1 position within each segment. Multiplied by 63
and passed to `ptau`, it produces smooth rises and falls.

## Palette 1: Rainbow

```glsl
vec3 palette1(float ci) {
  float t = ci / 256.0 * 4.0;
  float seg = floor(t);
  float f = fract(t);
  if (seg < 1.0) return vec3(ptau(f * 63.0), ptau(0.0), ptau(0.0));
  if (seg < 2.0) return vec3(ptau((1.0 - f) * 63.0), ptau(0.0), ptau(f * 63.0));
  if (seg < 3.0) return vec3(ptau(0.0), ptau(f * 63.0), ptau((1.0 - f) * 63.0));
  return vec3(ptau(f * 63.0), ptau(63.0), ptau(f * 63.0));
}
```

| Segment | Visual |
|---------|--------|
| 0 | Black → red |
| 1 | Red → purple (red fades, blue rises) |
| 2 | Purple → green (blue fades, green rises) |
| 3 | Green → white-ish (red + green at max, approaching white) |

This palette cycles through the colour wheel, giving the "rainbow" look.

## Palette 2: Grey

```glsl
vec3 palette2(float ci) {
  float t = ci / 256.0 * 4.0;
  float seg = floor(t);
  float f = fract(t);
  float v;
  if (seg < 1.0) v = ptau(0.0);
  else if (seg < 2.0) v = ptau(f * 63.0);
  else if (seg < 3.0) v = ptau((1.0 - f) * 63.0);
  else v = ptau(0.0);
  return vec3(v * 0.5, v * 0.5, v * 0.5);
}
```

All three channels are equal (`v * 0.5`), producing pure greyscale. Only
segments 1–2 have brightness variation; segments 0 and 3 are flat dark.
The `× 0.5` dims the palette for a moody, subdued look.

## Drop Transitions

The demo cycles through three palette sequences with dramatic "drop"
transitions. At specific sync frames, the plasma appears to fall off the
bottom of the screen:

```javascript
const SYNC_FRAMES = [0, 880, 1760, 2400];
```

| Sync frame | Palette | Time |
|------------|---------|------|
| 0 | Red (palette 0) | 0.0 s |
| 880 | Rainbow (palette 1) | 12.6 s |
| 1760 | Grey (palette 2) | 25.1 s |
| 2400 | Grey (continues) | 34.3 s |

When a sync frame is reached, the drop transition plays over 64 frames:

```javascript
if (seq > 0) {
  const dropFrame = frame - SYNC_FRAMES[seq];
  if (dropFrame < 64) {
    lc = Math.floor(dropFrame * dropFrame / 4 * 43 / 128 + 60);
    inDrop = true;
  }
}
```

The `dropFrame²` term makes the vertical offset accelerate **quadratically**
— the plasma starts sliding slowly, then rushes downward. This mimics
gravity: the pattern falls with increasing speed until it disappears off the
bottom edge. Then the new palette sequence fades in from black.

The fade logic controls brightness during transitions:

```javascript
let fadeFrac = 1.0;
if (seq === 0 && frame < 126) {
  fadeFrac = frame / 126;        // initial fade-in over ~1.8 seconds
} else if (inDrop) {
  fadeFrac = 0.0;                // black during drop
} else if (seq > 0 && seqFrame < 30) {
  fadeFrac = seqFrame / 30;      // fade-in after drop
}
```

## Colour Remapping Matrix

The three procedural palettes produce the classic colour schemes. But the
remastered variant offers 21 selectable themes — without defining 21 separate
palette functions. Instead, it uses a **3×3 colour remapping matrix**:

```glsl
color = clamp(mat3(uColorR, uColorG, uColorB) * color, 0.0, 1.0);
```

Each theme defines three RGB vectors that remap the red, green, and blue
channels:

```javascript
const PALETTES = [
  { name: 'Classic',    colorMap: [[1,0,0],       [0,1,0],       [0,0,1]]       },
  { name: 'Ember',      colorMap: [[1,0.35,0],    [0.7,0.8,0.1], [0.4,0.08,0.05]] },
  { name: 'Ocean',      colorMap: [[0.05,0.3,0.9],[0.1,0.8,0.5], [0.7,0.15,0.25]] },
  // ... 18 more themes
];
```

For "Classic", the matrix is the identity — red stays red, green stays green,
blue stays blue. For "Ocean", red input maps mostly to blue output (`0.9`),
green input maps to cyan (`0.8 green + 0.5 blue`), and blue input maps to
warm tones. The same plasma pattern, the same palette function, but
completely different colours.

This is equivalent to multiplying every pixel's RGB by a 3×3 matrix:

```
┌ outR ┐   ┌ colorR.r  colorG.r  colorB.r ┐   ┌ inR ┐
│ outG │ = │ colorR.g  colorG.g  colorB.g │ × │ inG │
└ outB ┘   └ colorR.b  colorG.b  colorB.b ┘   └ inB ┘
```

A single matrix multiply per pixel transforms the entire colour palette.
This is vastly more efficient than defining separate palette functions for
each theme.

## Hue Rotation

Beyond theme selection, the editor exposes a continuous **hue shift** control
(0–360°). This rotates all colours around the perceptual colour wheel:

```glsl
vec3 hueRotate(vec3 color, float angle) {
  float cosA = cos(angle);
  float sinA = sin(angle);
  mat3 rot = mat3(
    0.299 + 0.701*cosA + 0.168*sinA,
    0.299 - 0.299*cosA - 0.328*sinA,
    0.299 - 0.299*cosA + 1.250*sinA,
    0.587 - 0.587*cosA + 0.330*sinA,
    ...
  );
  return clamp(rot * color, 0.0, 1.0);
}
```

The magic numbers (0.299, 0.587, 0.114) are the **luminance weights** for
red, green, and blue — the standard NTSC/BT.601 coefficients. The matrix
rotates colours while preserving perceived brightness (a red and a green of
the same brightness stay equally bright after rotation). Without luminance
preservation, rotating hue would cause visible brightness flickering.

## Saturation Boost

The saturation control adjusts how vivid or muted the colours appear:

```glsl
vec3 boostSaturation(vec3 color, float amount) {
  float luma = dot(color, vec3(0.299, 0.587, 0.114));
  return clamp(mix(vec3(luma), color, 1.0 + amount), 0.0, 1.0);
}
```

This works by computing the **luminance** (perceived brightness) and then
interpolating between a greyscale version of the colour and the original:

- `amount = 0`: no change (mix factor = 1.0, full colour)
- `amount = 0.5`: extra vivid (mix factor = 1.5, colours pushed beyond original)
- `amount = -0.5`: desaturated (mix factor = 0.5, halfway to grey)

The `mix()` with a factor > 1.0 extrapolates beyond the original colour,
making it more saturated. The `clamp()` prevents values from going negative
or above 1.0.
